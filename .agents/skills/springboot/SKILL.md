# Spring Boot Skill

This skill provides guidance for working with the spring-projects/spring-boot repository. It helps AI coding agents understand the project structure, build system, and development workflow.

## Project Overview

Spring Boot is a framework for building production-grade Spring applications with minimal configuration. The repository contains the core framework, starters, auto-configuration, and various tooling (Maven/Gradle plugins, CLI, etc.).

## Repository Structure

- `spring-boot-project/` – main source code, subdivided into:
  - `spring-boot/` – core library (SpringApplication, etc.)
  - `spring-boot-autoconfigure/` – auto-configuration for many technologies
  - `spring-boot-starters/` – starter POMs
  - `spring-boot-actuator/`, `spring-boot-actuator-autoconfigure/`
  - `spring-boot-test/`, `spring-boot-test-autoconfigure/`
  - `spring-boot-devtools/`
  - `spring-boot-properties-migrator/`
  - `spring-boot-docs/`
  - and more.
- `spring-boot-tests/` – integration and smoke tests
- `spring-boot-system-tests/` – system tests
- Other modules: `spring-boot-tools/` (build plugins, loader tools), `spring-boot-cli/`, samples, etc.

## Build System

- Uses **Gradle** (Groovy DSL). Root `settings.gradle` includes all modules.
- Gradle wrapper: `gradlew` (Unix) / `gradlew.bat` (Windows).
- Java 17 is required for building (Spring Boot 3.x). Check `sourceCompatibility` in build scripts.
- Key tasks:
  - `./gradlew build` – compiles, runs tests, generates docs.
  - `./gradlew test` – run unit tests across all modules (or specific module: `:spring-boot-project:spring-boot-autoconfigure:test`).
  - `./gradlew check` – full validation (includes style checks, like Checkstyle).
  - `./gradlew publishToMavenLocal` – for local testing of dependency changes.
- Builds produce JARs (executable and normal), documentation, and can publish to Artifactory.

## Development Workflow

1. **Clone and set up**: `git clone https://github.com/spring-projects/spring-boot.git`
2. **Import into IDE**: The project has `.idea` (IntelliJ) configuration. Run `./gradlew build` first, then import as Gradle project.
3. **Make changes**: Follow existing coding conventions (Spring Java Format is enforced via Checkstyle and formatting plugins). Use `git commit` with conventional commit messages (see CONTRIBUTING.md).
4. **Test**: Always run tests in the affected module(s). Use `./gradlew :spring-boot-project:<module>:test` or `./gradlew test` for all.
5. **Run specific test class**: `./gradlew :spring-boot-project:spring-boot-autoconfigure:test --tests com.example.MyTest`
6. **Check style**: `./gradlew checkstyleMain checkstyleTest` or `./gradlew format` (if Spring Format plugin applied).
7. **Submit a PR**: Push to fork, open pull request against `main` branch.

## Key Conventions

- Use Java 17 language features where appropriate.
- Follow Spring Framework coding style: 120 char line limit, indent with 4 spaces, no tabs.
- Use `AssertJ` for assertions, `Mockito` for mocking in tests.
- Auto-configuration classes are annotated with `@AutoConfiguration` and registered via `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`.
- Configuration properties are bound via `@ConfigurationProperties` and metadata generated by `spring-boot-configuration-processor`.
- All public API changes should have corresponding documentation updates in `.adoc` files under `spring-boot-docs`.

## Common Tasks

- **Adding a new auto-configuration**:
  - Create auto-configuration class in `spring-boot-autoconfigure` module.
  - If needed, add a new starter in `spring-boot-starters`.
  - Register auto-configuration in `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`.
  - Add tests under `spring-boot-autoconfigure` test source set.
- **Adding a new endpoint** in Actuator: see existing endpoints in `spring-boot-actuator` module; follow `@Endpoint` and `@ReadOperation`/`@WriteOperation`.
- **Upgrading third-party dependencies**: Bump versions in `gradle.properties` or build scripts; ensure compatibility and run full build.

## Testing Strategies

- Unit tests use JUnit 5 (Jupiter). Test classes often annotated with `@SpringBootTest` for integration tests.
- For auto-configuration tests, use `ApplicationContextRunner` from `spring-boot-test-autoconfigure`.
- Smoke tests in `spring-boot-tests/spring-boot-smoke-tests/` cover end-to-end scenarios; these run against a specific version of Spring Boot.
